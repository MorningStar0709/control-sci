Regarding the collision checking in $\bar { \lambda } _ { \mathrm { f r e e } } ( \bar { e } _ { t r } , \bar { \eta } _ { r } )$ of the B-RRT algorithm, we check a finite number of samples around each point of the resulting trajectory $q _ { \mathrm { d } }$ for collision. We run B-RRT with 10 and 50 such samples and we compared the results to a standard geometric RRT algorithm in terms of time per number of nodes. The results for 30 runs of the algorithms are given in Fig. 6.4 for the four paths, in logarithmic scale. One can notice that the average nodes created do not differ significantly among the different algorithms. As expected, however, B-RRT requires more time than the standard geometric RRT algorithm, since it checks the extra samples in $\bar { \lambda } _ { \mathrm { f r e e } } ( \bar { e } _ { t r } , \bar { \eta } _ { r } )$ for collision. One can also notice that the time increases with the number of samples. However, more samples imply greater coverage of $\bar { \lambda } _ { \mathrm { f r e e } } ( \bar { e } _ { t r } , \bar { \eta } _ { r } )$ and hence the respective solutions are more likely to be complete with respect to collisions.
